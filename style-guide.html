Living documentation of design tokens, components, and patterns

Overview

This style guide documents the design system for exercAIse. All styles use CSS custom properties (design tokens) for consistency and easy theming including automatic dark mode support.

Design Principles

  • Token-based: All colors, spacing, and typography use CSS variables
  • Mobile-first: Responsive design starting from mobile viewport
  • Dark mode native: Automatic dark mode support via prefers-color-scheme
  • Accessible: WCAG AA compliant contrast ratios and focus states
  • Consistent: BEM naming conventions for predictable class names

Button Guidelines

  • Default: Standard actions (Save, Submit, Continue)
  • Primary: Most important action on a page (rare, use sparingly)
  • Secondary: Less important actions (Cancel, Back)
  • Danger: Destructive actions (Delete, Remove)
  • Minimum touch target: 44px × 44px for accessibility
  • Use clear, action-oriented labels ("Save Workout" not "OK")

Form Guidelines

  • Always include labels for accessibility
  • Use form__hint for helper text below inputs
  • Group action buttons in form-actions / form__actions container
  • Place destructive actions (Cancel) on left, primary on right
  • Minimum input height: 44px for touch targets

Naming Conventions

BEM Methodology

We use BEM (Block Element Modifier) for component class names:

Type Pattern Example Usage
Block .block-name .exercise-card Standalone component
Element .block__element .exercise-card__header Part of a block
Modifier .block--modifier .exercise-card--compact Variation of a block

Class Prefixes

Prefix Usage Example
l- Layout containers .l-container, .l-grid
(none) Utility classes (spacing, flex, typography) .flex, .mb-4, .text-center
is-, has- State classes .is-active, .has-error
js- JavaScript hooks (no styles) .js-toggle-menu

Note: Utility classes don't use the u- prefix for brevity and readability. This is a pragmatic exception to the prefix convention.

Do's and Don'ts

✓ Do

.exercise-card__header

.button--primary

.flex, .mb-4

Use descriptive, semantic names that describe purpose, not appearance.

✗ Don't

.bigRedButton

.div1

.blue-text

Avoid appearance-based names, vague names, or camelCase (use kebab-case).

Responsive Design

Breakpoints

Name Width Devices
Mobile 0 - 640px Phones (portrait)
Tablet 641px - 768px Phones (landscape), small tablets
Desktop 1024px+ Laptops, desktops

Mobile-First Approach

Start with mobile styles, then add complexity for larger screens:

Responsive Pattern
/* Mobile-first: start with smallest screen */
.component {
  padding: 12px;
  font-size: 14px;
}

/* Tablet and up */
@media (min-width: 641px) {
  .component {
    padding: 16px;
    font-size: 16px;
  }
}

/* Desktop and up */
@media (min-width: 1024px) {
  .component {
    padding: 20px;
    font-size: 18px;
  }
}

Touch Targets

  • Minimum 44px × 44px for all interactive elements (buttons, links)
  • Adequate spacing between touch targets (at least 8px)
  • Larger tap areas on mobile (use padding, not just visual size)

Accessibility Checklist

Core Requirements

Color & Contrast

  • ✓ Text contrast meets WCAG AA minimum (4.5:1 for normal text, 3:1 for large text)
  • ✓ Color is not the only means of conveying information
  • ✓ Focus indicators have 3:1 contrast against background
  • ✓ Dark mode maintains proper contrast ratios

Keyboard Navigation

  • ✓ All interactive elements accessible via keyboard (Tab, Enter, Space)
  • ✓ Logical tab order follows visual layout
  • ✓ Focus states clearly visible
  • ✓ No keyboard traps (user can escape from all components)
  • ✓ Skip links available for screen readers

Touch & Click Targets

  • ✓ Minimum 44px × 44px touch targets on mobile
  • ✓ Adequate spacing between interactive elements (8px minimum)
  • ✓ Buttons and links have visible hover/active states

Semantic HTML

  • ✓ Use semantic elements (button, nav, main, header, footer)
  • ✓ Forms have associated labels (for/id relationship)
  • ✓ Heading hierarchy is logical (h1 → h2 → h3)
  • ✓ Lists use ul/ol/li appropriately
  • ✓ Images have alt text

ARIA & Screen Readers

  • ✓ Use ARIA labels only when semantic HTML isn't sufficient
  • ✓ Form validation errors announced to screen readers
  • ✓ Loading states and dynamic content changes announced
  • ✓ Modal dialogs trap focus and have proper aria-modal

Common Patterns

✓ Accessible Button

<button type="button">
  Save Workout
</button>

Uses semantic button element with descriptive text.

✗ Inaccessible "Button"

<div onclick="save()">
  Save
</div>

Not keyboard accessible, no semantic meaning.

✓ Accessible Form Input

<label for="weight">
  Weight (lbs)
</label>
<input 
  type="number" 
  id="weight"
  aria-describedby="weight-hint"
>
<p id="weight-hint" class="form__hint">
  Enter weight used
</p>

Label associated, hint connected via aria-describedby.

✗ Inaccessible Input

<input 
  type="number" 
  placeholder="Weight"
>

No label, placeholder isn't a substitute for label.

Dark Mode

Dark mode is automatically applied based on system preference using prefers-color-scheme: dark.

Testing Dark Mode

  • macOS: System Preferences → General → Appearance → Dark
  • Windows: Settings → Personalization → Colors → Dark
  • Browser DevTools: Chrome/Firefox DevTools → Rendering → Emulate CSS media feature

Dark Mode Guidelines

  • All colors are automatically inverted via CSS custom properties
  • Text contrast meets WCAG AA standards in both modes
  • Don't hardcode colors - always use CSS variables
  • Test all components in both light and dark mode
  • Headings use #f8fafc (near-white) in dark mode for maximum readability
